# TeenC++ 质量保证流程框架 v1.0

## 一、流程分层架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     用户验收测试 (UAT)                           │
│                   准入标准: 全部门禁通过                         │
├─────────────────────────────────────────────────────────────────┤
│                     集成测试阶段                                 │
│         新功能集成验证 + 回归测试 + 性能基线                     │
├─────────────────────────────────────────────────────────────────┤
│                     单元测试阶段                                 │
│              模块级功能验证 + 边界条件覆盖                       │
├─────────────────────────────────────────────────────────────────┤
│                     开发自测阶段                                 │
│         代码审查 + 静态分析 + 开发者自测清单                     │
└─────────────────────────────────────────────────────────────────┘
```

## 二、各阶段质量门禁

### 2.1 开发自测门禁（Developer Self-Test Gate）

**触发时机**：功能编码完成，提交测试前

| 检查项 | 要求 | 验证方法 |
|--------|------|----------|
| **代码编译** | 0 error，警告需评估 | `dotnet build` |
| **代码规范** | 符合项目命名约定 | 代码审查清单 |
| **功能完整性** | 需求功能全部实现 | 需求追踪表核对 |
| **基础可运行** | 主流程能跑通 | 开发者手动验证 |
| **无害性** | 不破坏原有功能 | 核心场景 spot-check |

**必检清单**（针对本次发现的代码框缺失问题）：
- [ ] UI控件是否正确显示（Visibility、Location、Size）
- [ ] 数据绑定是否正确（数据流向、默认值）
- [ ] 用户交互是否响应（事件绑定、回调函数）
- [ ] 新旧格式兼容性（L01/L02/L03都验证）

**产出物**：`DEV_TEST_REPORT.md` 开发者自测报告

---

### 2.2 单元测试门禁（Unit Test Gate）

**触发时机**：模块功能稳定后

| 维度 | 标准 | 工具 |
|------|------|------|
| **覆盖率** | 核心逻辑 ≥80%，UI交互 ≥60% | VSTest + Coverlet |
| **通过率** | 100%通过，0 flaky test | `dotnet test` |
| **边界覆盖** | 空值、异常、极值都有用例 | 代码审查 |
| **依赖隔离** | 外部依赖已Mock | Moq验证 |

**产出物**：测试报告含覆盖率数据

---

### 2.3 回归测试门禁（Regression Test Gate）

**触发时机**：每次提交前 + 版本发布前

**回归范围矩阵**：

| 改动类型 | 必测范围 | 抽样范围 |
|----------|----------|----------|
| UI组件新增 | 所有课程加载、显示 | 边缘分辨率适配 |
| 数据模型变更 | 新旧格式课程各1个 | 极端数据量 |
| 数据库变更 | Schema迁移、数据兼容 | 性能基准 |
| 业务流程变更 | 完整通关流程 | 异常中断恢复 |

**v1.2.0 回归测试用例集**：

```
TC-REG-01: L01 旧格式课程完整通关
  1. 启动应用
  2. 进入第1课
  3. 验证：代码编辑器显示正常
  4. 验证：步骤说明正常显示
  5. 修改代码并检查
  6. 生成项目并验证文件
  预期：全流程通过，XP正确累加

TC-REG-02: L02 旧格式课程完整通关
  (同TC-REG-01)

TC-REG-03: L03 新格式课程知识阶段
  1. 启动应用
  2. 进入第3课
  3. 验证：显示阶段选择界面
  4. 进入知识阶段
  5. 验证：场景、概念、测验、填空各环节正常
  预期：知识阶段XP正确累计

TC-REG-04: L03 新格式课程实践阶段
  1. 完成知识阶段或跳过
  2. 进入实践阶段
  3. 验证：代码编辑器显示正常
  4. 验证：检查、生成按钮可用
  预期：代码编辑功能正常

TC-REG-05: 阶段进度持久化
  1. 完成L03知识阶段
  2. 关闭应用
  3. 重新启动
  4. 再次进入L03
  预期：显示已完成的阶段，无需重复
```

**产出物**：`REGRESSION_TEST_REPORT.md`

---

### 2.4 集成测试门禁（Integration Test Gate）

**触发时机**：功能模块全部完成，准备用户测试

| 检查项 | 验收标准 |
|--------|----------|
| **端到端流程** | 从启动到通关的完整流程可跑通 |
| **数据一致性** | 课程数据→UI显示→检查结果→XP奖励一致 |
| **异常处理** | 文件缺失、网络中断、用户取消都有处理 |
| **并发安全** | 多窗体切换、快速操作无崩溃 |
| **资源释放** | 关闭后无内存泄漏、文件句柄泄漏 |

## 三、用户测试准入标准（User Testing Entry Criteria）

### 3.1 进入Beta测试的最低要求

```
必须全部满足（P0）：
□ 编译0 error
□ 单元测试 100% 通过（当前基准8/8）
□ 核心功能回归测试 100% 通过
□ 需求功能实现完整（功能清单核对）
□ 无明显UI错位或遮挡
□ 应用启动到主界面 ≤ 3秒

应该满足（P1）：
□ 可空性警告 < 50个
□ 代码覆盖率达到基准线
□ 边界情况已处理（空目录、缺失文件等）

建议满足（P2）：
□ UI自动化测试覆盖主流程
□ 性能基准测试通过
```

## 四、本次v1.2.0问题复盘
