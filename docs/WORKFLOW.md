# TeenC++ 教学系统 - 标准化工作流程

## 概述

本文档定义 TeenC++ 教学系统从需求到发布的完整工作流程，确保项目质量、可追溯性和团队协作效率。

---

## 一、流程总览

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   需求阶段   │ → │   设计阶段   │ → │   开发阶段   │ → │   交付阶段   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
       │                  │                  │                  │
       ▼                  ▼                  ▼                  ▼
  [需求收集]          [产品方案]          [编码实现]          [构建发布]
  [可行性分析]        [技术架构]          [代码审查]          [版本管理]
       │                  │                  │                  │
       ▼                  ▼                  ▼                  ▼
  输出：PRD           输出：设计文档       输出：可运行程序     输出：Release
       +               + 接口契约         + 单元测试报告      + 更新日志
  审核点：必要性      审核点：合理性      审核点：规范性      审核点：完整性
```

## 二、阶段详解

### Phase 1: 需求阶段

#### 2.1.1 触发条件
- 新功能需求
- Bug 修复
- 性能优化
- 课程内容新增

#### 2.1.2 活动清单
| 活动 | 负责人 | 输出物 | 检查点 |
|------|--------|--------|--------|
| 需求收集 | 产品经理/用户 | 需求描述 | 需求是否清晰？ |
| 可行性分析 | 技术负责人 | 可行性报告 | 技术可实现？时间允许？ |
| 优先级排序 | 项目负责人 | 优先级标签 | P0/P1/P2？ |

#### 2.1.3 审核标准
- [ ] 需求与项目目标一致
- [ ] 目标用户明确
- [ ] 验收标准可量化
- [ ] 技术风险已识别

#### 2.1.4 输出模板
使用 `templates/PRD_TEMPLATE.md` 编写产品需求文档

### Phase 2: 设计阶段

#### 2.2.1 产品设计
| 活动 | 输出物 | 检查点 |
|------|--------|--------|
| 功能设计 | 功能流程图 | 流程是否完整？ |
| UI/UX设计 | 界面原型 | 用户体验是否友好？ |
| 交互设计 | 交互说明 | 操作是否直观？ |

#### 2.2.2 技术架构设计
| 活动 | 输出物 | 检查点 |
|------|--------|--------|
| 架构设计 | 架构文档 | 是否符合现有架构？ |
| 接口设计 | API契约 | 接口是否稳定？ |
| 数据设计 | 数据模型 | 是否满足需求？ |

#### 2.2.3 审核节点：设计评审会

**评审参与者**：
- 产品负责人（功能正确性）
- 技术负责人（技术合理性）
- 测试负责人（可测试性）

**评审检查清单**：
- [ ] **功能完整性**：是否覆盖所有需求？
- [ ] **技术可行性**：技术方案是否可实现？
- [ ] **风险评估**：潜在风险及应对措施？
- [ ] **性能影响**：对现有性能的影响？
- [ ] **兼容性**：是否兼容现有课程数据？
- [ ] **可扩展性**：未来扩展是否方便？

**评审结论**：
- ✅ 通过 → 进入开发阶段
- ⚠️ 修改后通过 → 指定修改项，修改后重新评审
- ❌ 不通过 → 重新设计

### Phase 3: 开发阶段

#### 2.3.1 编码规范

**C# 编码规范**：
- 遵循 [Microsoft C# Coding Conventions](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/coding-style/coding-conventions)
- 使用 VS 代码分析器，保持 0 错误，最少警告
- 关键逻辑必须添加注释
- 公共 API 必须添加 XML 文档注释

**提交规范**：
```
<type>(<scope>): <subject>

<body>

<footer>
```

类型说明：
- `feat`: 新功能
- `fix`: Bug修复
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具

#### 2.3.2 代码审查（Code Review）

**审查清单**：
- [ ] **功能正确性**：代码是否实现了设计？
- [ ] **代码规范**：是否符合编码规范？
- [ ] **异常处理**：错误处理是否完善？
- [ ] **性能**：是否存在性能问题？
- [ ] **安全性**：是否存在安全隐患？
- [ ] **可测试性**：是否方便测试？

**审查流程**：
1. 开发者提交 Pull Request / 变更说明
2. 指定审查人（至少1人）
3. 审查人提出意见
4. 开发者修改或回应
5. 审查通过后合并

#### 2.3.3 单元测试

**测试要求**：
- 核心算法必须有单元测试
- 检查规则必须有测试用例
- 测试覆盖率目标：> 70%

**测试命名规范**：
```csharp
[TestMethod]
public void MethodName_Scenario_ExpectedResult()
{
    // Arrange
    // Act
    // Assert
}
```

### Phase 4: 测试阶段

#### 2.4.1 测试类型

| 测试类型 | 负责人 | 检查点 |
|----------|--------|--------|
| 单元测试 | 开发者 | 代码逻辑正确 |
| 集成测试 | 测试人员 | 模块间协作正常 |
| 功能测试 | 测试人员 | 符合需求文档 |
| 回归测试 | 自动化/测试人员 | 不影响现有功能 |
| 用户验收测试 | 产品经理/用户 | 满足用户期望 |

#### 2.4.2 测试检查清单

**功能测试清单**（参考 `docs/TEST_PLAN.md`）：
- [ ] 程序启动正常
- [ ] 主界面显示正确
- [ ] 课程列表加载正确
- [ ] 课程界面功能完整
- [ ] 代码编辑器工作正常
- [ ] 代码检查（通过/失败）场景
- [ ] 导师放行功能
- [ ] Dev-C++ 项目生成
- [ ] 通关解锁机制
- [ ] 数据持久化

**测试覆盖率要求**（质量门禁）：

| 类型 | 目标 | 最低要求 | 门禁 |
|------|------|----------|------|
| **总覆盖率** | ≥ 80% | ≥ 70% | ❌ 阻塞发布 |
| **核心模块** (Core.Checkers) | ≥ 90% | ≥ 80% | ❌ 阻塞发布 |
| **Services** | ≥ 70% | ≥ 60% | ⚠️ 需批准 |
| **UI** | ≥ 50% | ≥ 40% | ⚠️ 需批准 |

**覆盖率统计命令**：
```bash
# 使用coverlet生成覆盖率报告
dotnet test TeenCppEdu.Tests /p:CollectCoverage=true /p:CoverletOutputFormat=cobertura
# 或
dotnet test --collect:"XPlat Code Coverage"
```

**兼容性测试**：
- [ ] Windows 10/11
- [ ] .NET Framework 4.8
- [ ] Dev-C++ 集成

#### 2.4.3 用户测试版本要求（UAT/Beta）

提交给用户测试的版本必须满足以下质量标准：

**1. 日志模块（强制）**

| 要求 | 说明 |
|------|------|
| **日志开启** | Release版本默认开启日志记录 |
| **日志级别** | 至少记录Info及以上级别 |
| **日志内容** | 启动日志、错误日志、关键操作日志 |
| **日志位置** | `%AppData%/TeenCppEdu/logs/` |
| **日志保留** | 保留最近7天，单文件最大10MB |

**日志检查清单**：
- [ ] 程序启动时记录版本、环境信息
- [ ] 异常/错误自动记录（包含堆栈）
- [ ] 用户关键操作记录（进入课程、代码检查等）
- [ ] 日志文件自动生成，不依赖外部配置

**2. 测试覆盖证明**

提交用户测试前必须提供：
- [ ] 单元测试覆盖率报告（达到门禁要求）
- [ ] 功能测试用例执行记录（通过率100%）
- [ ] 已知问题清单（Bug跟踪表）
- [ ] 回归测试通过证明

**3. 版本标识**

用户测试版本必须明确标识：
```
版本号格式: X.Y.Z-beta.N 或 X.Y.Z-uat.N
示例: 1.0.0-beta.1, 1.1.0-uat.2
```

在程序界面和日志中显示：
- [ ] 当前版本号（包含beta/uat标识）
- [ ] 构建时间
- [ ] 版本类型（DEBUG/RELEASE/BETA）

#### 2.4.3 Bug 管理

**严重程度定义**：
- **P0-致命**：系统崩溃，数据丢失，无法使用
- **P1-严重**：主要功能不可用，有明显 workaround
- **P2-一般**：次要功能问题，影响体验
- **P3-轻微**：UI瑕疵，建议改进

**修复时限**：
- P0: 立即修复
- P1: 24小时内
- P2: 下个迭代
- P3: 排期处理

### Phase 5: 构建与发布

#### 2.5.1 构建流程

```bash
# 1. 清理
msbuild TeenCppEdu.sln /t:Clean

# 2. 还原依赖
nuget restore TeenCppEdu.sln

# 3. 编译
msbuild TeenCppEdu.sln /p:Configuration=Release

# 4. 运行测试
dotnet test

# 5. 打包
# 复制 exe + 依赖 + 课程数据到发布目录
```

#### 2.5.2 质量门禁

**构建失败条件（阻塞发布）**：
- [ ] 编译错误 > 0
- [ ] 编译警告 > 100（需清理）
- [ ] 单元测试失败 > 0
- [ ] 测试覆盖率 < 70%（总覆盖率）
- [ ] 核心模块覆盖率 < 80%（Core.Checkers等）
- [ ] 关键功能测试失败 > 0
- [ ] 日志模块缺失或无法正常工作（用户测试版本）

**质量门禁检查清单**：
```powershell
# 编译检查
dotnet build -c Release
# 预期: 0错误

# 单元测试
dotnet test
# 预期: 100%通过

# 覆盖率检查（如已配置）
dotnet test /p:CollectCoverage=true
# 预期: ≥70%总覆盖率，≥80%核心模块

# 日志验证（用户测试版本）
# 1. 运行程序，检查日志目录是否自动创建
# 2. 检查日志文件是否包含启动信息
# 3. 触发异常，检查错误日志是否记录
```

#### 2.5.3 版本管理

**版本号格式**：`主版本.次版本.修订号.构建号`
示例：`1.2.3.0`

**标签定义**：
- **主版本**：重大功能更新，架构变更
- **次版本**：新功能发布
- **修订号**：Bug 修复
- **构建号**：CI 构建序号

**发布说明模板**（`templates/RELEASE_NOTES_TEMPLATE.md`）：
```markdown
## v1.2.0 (2026-03-21)

### 新增功能
- 功能A描述
- 功能B描述

### 修复问题
- 修复问题C
- 修复问题D

### 已知问题
- 已知问题E（计划v1.2.1修复）

### 兼容性
- 需要 .NET Framework 4.8
- 支持 Windows 10/11
```

## 五、迭代节奏

### 5.1 迭代周期

**建议**：2周一个迭代（Sprint）

**节奏安排**：
- 第1天：迭代计划会（确定本次迭代任务）
- 每天：站会（同步进度和问题）
- 第10天：功能冻结（只修Bug，不加功能）
- 第13-14天：测试、修复、发布

### 5.2 会议制度

| 会议 | 频率 | 参与者 | 时长 |
|------|------|--------|------|
| 迭代计划会 | 每迭代1次 | 全员 | 1小时 |
| 每日站会 | 每天 | 开发/测试 | 15分钟 |
| 设计评审会 | 按需 | 相关人 | 1小时 |
| 代码评审 | 按需 | 相关人 | 30分钟 |
| 迭代回顾 | 每迭代1次 | 全员 | 30分钟 |
