# TeenC++ 发布政策

## 核心原则

> **只发布正式版本到 GitHub**

---

## 禁止事项

❌ **禁止以下操作**：

1. 创建 Beta/Alpha/RC 的 Git 标签并推送
2. 在 GitHub Releases 页面发布非正式版本
3. 将测试版本推送给最终用户
4. 在 Release Note 中使用 "测试版"、"预览版" 等字样

---

## 允许事项

✅ **只允许以下操作**：

| 版本类型 | 格式 | 操作 |
|----------|------|------|
| 正式版 | `vX.Y.Z` | 创建标签 + GitHub Release |
| 补丁版 | `vX.Y.Z` | 同上 |
| 主版本 | `vX.0.0` | 同上 |

---

## 开发测试版本如何处理

### Beta 版本测试流程

```
开发分支 → 本地构建 → 本地测试
                ↓
            ❌ 不创建标签
            ❌ 不推送到 GitHub
            ❌ 不创建 Release
                ↓
            修复 Bug
                ↓
            测试通过
                ↓
        合并到 main
                ↓
        发布正式版 vX.Y.Z
```

### 为什么这样规定

1. **用户体验**：只提供稳定可用的版本
2. **减少混淆**：Release 页面只有正式版本
3. **降低风险**：避免用户下载到有Bug的版本
4. **维护简单**：不需要管理大量废弃的 beta 版本

---

## 强制执行

### 使用 release.sh 脚本

```bash
# 正确 - 会成功
./tools/release.sh v1.3.0

# 错误 - 会被拒绝
./tools/release.sh v1.3.0-beta.1
# 输出: 错误: 版本号格式不正确
```

### 手动检查清单

发布前必须确认：
- [ ] 版本号格式为 `vX.Y.Z`
- [ ] 没有使用 `beta`、`alpha`、`rc` 等字样
- [ ] 所有测试通过
- [ ] 用户验收完成

## 违规处理

如果不小心误发了 beta 版本到 GitHub：

1. **删除 Release**
   ```bash
   gh release delete v1.3.0-beta.1 --repo kaige2005/teenagercpp
   ```

2. **删除标签**
   ```bash
   git push origin --delete v1.3.0-beta.1
   git tag -d v1.3.0-beta.1
   ```

3. **通知用户**（如果已有人下载）
   - 在 Release 页面添加弃用声明
   - 引导用户使用正式版

---

## 例外情况

**没有例外**

任何情况下都不应将非正式版本发布到 GitHub。

如果确实需要外部测试：
- 个别文件分享（非 Release）
- 邮件发送
- 其他渠道

但绝不使用 GitHub Release 功能。
